# 如何使用任務卡

## 時間

Phase 0 會用到。

## 先做這三步

1. 打開台上或 GitHub Issue 指到的 Phase 0 任務卡。
2. 先看「你現在拿到的來源」和「你要做什麼」。
3. 開始前說清楚：要改哪裡、成果會從哪個畫面看到、哪些事現在不做。

Phase 0 請先讀 `docs/output-paths.md`，確認畫面要從哪個檔案進入。

## 共通規則

- 可展示成果要能從 GitHub Pages 首頁看到或操作。
- 前端成果要接到 `src/app/App.tsx` 或它匯入的 component。
- Phase 0 原始資訊在 `src/fixtures/phase-0/`。
- 不要把未整理資料搬進 shared fixtures 假裝乾淨。
- 不新增後端、資料庫、登入、Docker、爬蟲、地圖 SDK、外部 API 或安全掃描工具。

更多路徑規則看 `docs/output-paths.md`。

## 常見詞

| 詞                    | 這裡的意思                         |
| --------------------- | ---------------------------------- |
| fixture               | repo 裡的假資料                    |
| 原始資訊 / 未整理資料 | 還不能直接相信或直接行動的資料     |
| 整理後資料            | 經過人類檢查後，格式較穩定的資料   |
| 資料格式              | 資料欄位和狀態規則                 |
| 轉換器                | 把一種資料格式轉成另一種的小函式   |
| 畫面入口              | demo 從哪個檔案一路顯示出來        |

## 問 Coding Agent 前

先要求 Agent 回答：

1. 要改哪些檔案？
2. demo 畫面從哪個檔案進入？
3. 會新增或更新什麼測試？若本階段不需要，寫「本階段不新增測試」。
4. 哪些成果只屬於文件或測試？
5. 不會做哪些超出範圍的事？

等小組確認後再讓 Agent 改檔。

## 停止條件

時間到就停止加功能，改成：

- 確認 demo 可見
- 補 observation 與 AI log
- 跑 `pnpm build`；sprint 結束或交接前盡量跑 `pnpm check`
- 若檢查沒過，記錄指令、錯誤和原因
- commit

不熟 Git 時先問組員或 mentor，不要用會刪掉別人改動的指令。
