# Phase 0 Observations

## 我們做出了什麼？

- 首頁已有 Phase 0 原始資訊列表，並能進入整理工作台。
- 工作台提供畫面狀態中的整理草稿，可以建立、編輯、刪除、重設。
- 草稿預設保守標示為需要人工確認，避免把未確認資訊當成已確認事實。
- 工作台可以用需求與地點候選標籤篩選，但這只是協助檢視，不是正式分類結果。

## 哪些資料讓 agent 很容易整理？

- 有明確時間、來源描述與限制的資訊比較容易被整理成候選草稿，例如現場更新有寫剩餘物資、下一次盤點時間或「不要送到此處」。
- 但這些仍然只是候選整理，因為 `verificationStatus` 仍是 `needs_review` 或 `unverified`。

## 哪些資料讓 agent 亂補或誤判？

- 只有模糊地點的資訊容易被 agent 補成看似明確的位置，例如「老雜貨店後面」不能被改寫成可派工地址。
- 只有社群貼文或群組轉述的資訊容易被誤當成已確認需求。
- 有截圖或過期名單時，agent 可能忽略「是哪一天」或「今天是否仍有效」。

## 哪些資訊不能直接變成任務？

- 缺少明確位置、當事人同意、現場安全狀態或最新盤點的資訊不能直接變成志工任務。
- 有互相矛盾說法的資訊不能直接派人，例如同一地點同時有人說可集合、有人說淹水不適合停留。
- 來自家屬、社群留言、志工轉述的資訊，需要先確認當事人狀況與公開資訊範圍。

## 哪些資訊的操作者不是當事人？

- 志工代長者轉述的資訊需要確認長者是否同意公開完整地址。
- 外地家屬來電表示親友疑似需要協助，但家屬不在現場，也無法確認位置。
- 社群貼文、群組留言、截圖轉傳都可能不是第一手資料。

## 我們剛剛偷偷假設了什麼？

- 假設文字中提到的地點足以讓志工找到現場，但原文常常不夠精確。
- 假設更新時間等於資訊仍有效，但物資、道路、集合點狀態可能很快改變。
- 假設「官方公告」來源一定可靠，但原文仍可能只是截圖，日期與同步狀態未確認。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要更清楚的查核流程，定義誰可以把資訊從待人工確認改成已確認。
- 需要保留原文依據，並把推測和原文內容分開。
- 需要讓每筆候選任務記錄「為什麼不能直接行動」。

## 我們需要更清楚資料格式的地方

- 需要分開記錄來源類型、回報者角色、當事人是否同意公開資訊。
- 需要記錄資訊有效時間，而不是只記錄更新時間。
- 需要記錄地點精確度，避免把模糊地點當成完整地址。
- 需要把「需求類型」與「地點單位」標成候選狀態，避免篩選結果被誤讀為已確認。

## 我們需要人工確認的地方

- 查核狀態仍是 `needs_review` 或 `unverified` 的所有資訊。
- 非當事人轉述、社群貼文、截圖、過期名單。
- 看起來可以形成任務，但缺少地點、時間、同意、現場安全或負責角色的資訊。
